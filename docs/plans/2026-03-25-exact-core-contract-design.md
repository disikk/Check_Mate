# Exact Core Contract Design

## Проблема

`tracker_parser_core` уже реализует значимую часть replay-grade exact-core логики, но ее контракт сейчас распределен по коду и тестам. Из-за этого часть семантики существует как "знание из реализации": какие инварианты считаются обязательными, что именно означает `uncertain`, как трактуются forced all-in, какие KO-поля являются текущим v1-контрактом, а какие только подготовкой к v2. Без явного документа любое следующее изменение в parser/normalizer/pot-resolution рискует менять поведение неосознанно.

## Цель

Зафиксировать текущий exact-core контракт в одном документе, привязать его к существующим структурам и закрыть недостающие тестовые пробелы так, чтобы каждое ключевое правило normalizer-а было защищено хотя бы одним явным тестом.

## Варианты

### 1. Только документ без изменений тестов

Плюсы:
- быстро;
- минимальный риск изменения кода.

Минусы:
- часть контракта останется недоказанной тестами;
- документ будет сильнее зависеть от "доверия к текущей реализации".

### 2. Документ + точечное добивание тестового покрытия

Плюсы:
- контракт становится проверяемым, а не только описанным;
- лучше соответствует Red/Green подходу;
- снижает риск случайной регрессии в следующих фазах.

Минусы:
- чуть дольше первой итерации;
- нужно аккуратно выбрать только реально недостающие тесты.

### 3. Сразу документ + большой рефакторинг surface-моделей

Плюсы:
- можно сразу переделать спорные места.

Минусы:
- нарушает задачу `P0-01`, которая должна сначала зафиксировать поведение;
- повышает риск смешать freeze контракта с функциональными изменениями.

## Выбранный подход

Выбран вариант 2: сначала документируем текущий контракт, затем точечно доводим тесты там, где ключевое правило есть в коде, но еще не закреплено явной проверкой.

## Дизайн

### 1. Источник истины для контракта

Добавляем `backend/docs/exact_core_contract.md` как канонический документ exact-core уровня. Он описывает:
- surface входа и выхода (`CanonicalParsedHand`, `NormalizedHand`);
- инварианты сохранения фишек и банков;
- pot slicing и eligibility;
- actor-order и legality;
- forced all-in и return-uncalled;
- snapshot semantics;
- текущую `KO semantics v1`;
- целевую `KO semantics v2` как явно не реализованный future contract;
- uncertainty / inconsistent contract.

### 2. Границы P0-01

В рамках `P0-01` мы не меняем бизнес-семантику. Разрешены только:
- документирование уже существующего поведения;
- новые/уточненные тесты;
- минимальные кодовые правки, если без них нельзя сделать контракт тестируемым или однозначным.

Не входят в задачу:
- новый parser grammar;
- изменение KO semantics;
- новый winner evidence model;
- typed uncertainty refactor.

### 3. Тестовая стратегия

Повторно используем существующий контур:
- `tests/hand_normalization.rs`;
- `tests/positions.rs`;
- при необходимости `tests/fixture_parsing.rs`.

Для каждого раздела контракта указываем защищающий тест. Если для правила нет достаточно явного теста, добавляем его по TDD.

### 4. Архитектурные обновления

В `CLAUDE.md` добавляем ссылку на exact-core contract как на канонический backend reference для parser/normalizer/pot-resolution semantics.

## Критерии готовности

- есть `backend/docs/exact_core_contract.md`;
- документ покрывает весь текущий `P0-01` scope;
- для каждого ключевого инварианта есть явная ссылка на тест;
- `CLAUDE.md` отражает новый архитектурный reference;
- релевантные тесты прогнаны после изменений.
